# EventBus 逻辑说明（事件驱动专版）

这份文档改为集中讲 `EventBus` 逻辑，目标是让事件驱动同学快速理解这套架构并直接接手开发。

## 1. 设计定位

当前 `dev_test` 的事件系统是“轻量事件总线”模型：

- `Controller` 负责发布核心游戏事件。
- `MainUI` 负责订阅事件并更新界面。
- `BasicEventLoop` 负责驱动循环，并发布循环生命周期事件。

事件系统文件：`core/events.py`

## 2. 总体架构符号图

```text

┌────────────────────┐          publish(...)          ┌────────────────────┐
│ BasicEventLoop     │ ─────────────────────────────> │ EventBus            │
│ - LOOP_STARTED     │                                │ - handlers[event]   │
│ - LOOP_STOPPED     │ <───────────────────────────── │ - history           │
└─────────┬──────────┘      subscribe(handler)        └───────┬────────────┘
					│                                                      │
					│ step_handler = Controller.run_round                  │ dispatch(event)
					v                                                      v
┌────────────────────┐          publish(...)          ┌────────────────────┐
│ Controller         │ ─────────────────────────────> │ MainUI              │
│ - GAME_LOADED      │                                │ - _on_event_*       │
│ - ROUND_STARTED    │                                │ - 棋盘/日志刷新      │
│ - ROUND_FINISHED   │                                │ - 状态区更新         │
│ - GAME_OVER        │                                │                      │
└────────────────────┘                                └────────────────────┘

数据源分支（影响 Controller 发布内容）:
runtime_env (Environment.step)  或  mock(rounds[current_round])
```

## 3. 回合时序符号图（当前实现）

```text
UI点击开始
	 |
	 v

	 |
	 v
Controller.run_round
	 |
	 +--> EventBus.publish(ROUND_STARTED)
	 |
	 +--> 执行回合
	 |    - runtime: environment.step()
	 |    - mock: 读取 rounds[current_round]
	 |

	 |

	 |

	 |
	 +--> _on_event_round_started / _on_event_round_finished
	 +--> 刷新棋盘和右侧日志
	 +--> 安排下一次 tick（1000ms）
```

## 4. 当前事件定义与职责

当前 `EventType`（在 `core/events.py`）：

- `LOOP_STARTED`：循环开始（来自 `BasicEventLoop`）。
- `LOOP_STOPPED`：循环结束（来自 `BasicEventLoop`）。
- `GAME_LOADED`：数据加载完成（来自 `Controller.load_game_data`）。
- `ROUND_STARTED`：回合开始（来自 `Controller.run_round`）。
- `ROUND_FINISHED`：回合结束（来自 `Controller.run_round`）。
- `GAME_OVER`：对局结束（来自 `Controller.run_round`）。
- `ACTION_SUBMITTED`：预留（当前基本未使用）。
- `ERROR`：预留（当前可继续完善）。

## 5. 当前实现的关键逻辑（简版）

### 5.1 EventBus

```text
subscribe(event_type, handler)
	-> handlers[event_type].append(handler)

publish(event_type, payload)
	-> 构造 GameEvent
	-> history.append(event)
	-> for handler in handlers[event_type]: handler(event)
```

### 5.2 BasicEventLoop

```text
run_steps(step_handler)
	-> publish(LOOP_STARTED)
	-> while running:
			 should_continue = step_handler()
			 if False: break
	-> publish(LOOP_STOPPED)
```

### 5.3 Controller 事件发布点

```text
load_game_data      -> publish(GAME_LOADED)
run_round(开始前)   -> publish(ROUND_STARTED)
run_round(结束后)   -> publish(ROUND_FINISHED)
run_round(游戏结束) -> publish(GAME_OVER)
```

## 6. 给事件驱动同学的直接任务（按事件系统本身）

### 6.1 必做增强

- 补全 `ERROR` 事件链路：
	- `Controller` 捕获异常后统一 `publish(ERROR, {...})`。
	- `MainUI` 统一监听并渲染错误级别信息。

- 启用 `ACTION_SUBMITTED`：
	- 在动作提交节点发布（runtime/mock 都要有）。
	- payload 至少包含：`source`, `round_number`, `actor`, `action_type`。

- 统一 payload 字段规范：
	- 建议固定键：`source`, `round_number`, `is_game_over`, `message`。

### 6.2 建议增强

- 增加事件取消订阅能力：`unsubscribe(event_type, handler)`。
- 增加安全分发：单个 handler 抛异常时不中断其他 handler。
- 给 `history` 增加查询过滤（按事件类型/最近 N 条）。

## 8. 一句话总结

你写的这套架构本质是“`Controller` 产出事件，`EventBus` 分发事件，`MainUI` 消费事件”；后续事件驱动开发就围绕“补事件类型、补发布点、统一 payload、强化分发鲁棒性”四件事展开。
