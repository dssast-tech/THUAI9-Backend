## 2. 法术统一玩法设计（建议作为实现规范）

## 2.1 统一术语

- 锁定法术：必须选定具体目标单位（单体）。
- 非锁定法术：选择一个区域，区域内目标按规则筛选并结算。
- 延时法术：施放成功后不立即生效，进入延时队列，寿命归零时触发。
- 范围法术：本质是区域结算，可与锁定/非锁定组合，但通常与非锁定配合。

## 2.2 统一施法流程（建议）

1. 输入合法性检查：施法者、法术、目标/区域、回合状态。
2. 资源检查：`AP >= 1` 且 `SP >= spellCost`。
3. 范围检查：
   - 锁定法术：施法者到目标距离 `<= spell.range`。
   - 非锁定法术：施法者到施法中心距离 `<= spell.range`。
4. 命中/发动判定（若此法术定义需要）：
   - 推荐统一使用 d20 判定模型（见 2.3）。
5. 若延时法术：加入延时队列并扣资源，结束本次施法。
6. 若瞬发法术：筛选作用目标并应用效果。
7. 记录日志与结果（命中、伤害、治疗、死亡、移除、剩余资源）。

## 2.3 建议公式（技能法术）

> 文档已有方向，但后端尚未完整落地，建议统一如下。

### 2.3.1 发动/命中公式

- 攻击投掷：

  `spell_throw = d20 + INT_mod + SpellPower`

- 防御阈值：

  `spell_def = target.magic_resist`

- 命中条件：

  `spell_throw > spell_def`

- 大成功/大失败：
  - `d20 = 1`：必失败
  - `d20 = 20`：必成功

- 非锁定法术额外门槛（按玩法文档）：

  `spell_throw >= 2.5 * SpellPower`

  注：该门槛用于“发动成功”判定，可先判发动，再判目标命中。

### 2.3.2 伤害/治疗公式

- 伤害法术建议：

  `raw = spell.baseValue (+ crit bonus if any)`

  `real = max(0, raw - target.magic_resist)`（魔法伤害）

- 治疗法术建议：

  `heal_real = min(target.max_health - target.health, spell.baseValue)`

## 2.4 消耗规则建议

- 普通技能法术：成功施法后扣 `1 AP` 与 `spellCost SP`。
- 延时法术：加入队列时扣资源；触发时不重复扣。
- 上限约束：单次 `spellCost` 原则上不超过 2（按玩法文档）。

## 2.5 延时机制建议

- 入队结构包含：施法者、法术模板、目标信息、剩余寿命、创建回合、所属阵营。
- 回合末统一处理：寿命-1，`lifespan==0` 触发并移除。
- 触发时目标重新判定策略需固定：
  - 方案 A：按“当前在区域内”判定（动态）
  - 方案 B：按“施法时命中列表”判定（静态）

建议优先方案 A，和现有环境值/区域法术语义更一致。

## 2.6 范围法术关键点（重要）

- 范围中心是“施法点”而非目标单位本身。
- 目标筛选必须明确敌我关系：
  - Damage / Debuff：默认只作用敌方。
  - Heal / Buff：默认只作用友方。
  - Move：默认只作用施法者或指定单体。
- 同阵营误伤是否允许必须写死，否则会出现测试结果不一致。
- 多目标结算顺序应固定（按行动队列或坐标排序）以保证可复现日志。

## 3. 对照：玩法文档 vs 后端当前实现

## 3.1 核心差异总览

### A. 命中公式与发动判定

- 玩法文档：技能法术应有投掷逻辑（`d20 + INT_mod + 法术强属性`）且非锁定有额外门槛。
- 后端当前：多数法术在 `executeSpell/ApplySpellEffect` 中直接按 `baseValue`结算，缺少完整命中公式。

影响：法术实测表现会偏“稳定命中”，与玩法文档不一致。

### B. 伤害抗性结算

- 玩法文档：法术应受法术豁免值约束（命中/减伤语义明确）。
- 后端当前（C# env）：Damage 分支直接 `health - baseValue`，未显式按魔抗减伤。

影响：法术伤害偏高，且不同防具对法术差异体现不足。

### C. 范围检查语义

- 玩法期望：检查“施法者到施法点/目标”的施法范围。
- C# env 当前 `IsInSpellRange(target, targetArea)` 更像“目标到区域中心”，语义偏离常见施法范围定义。

影响：某些边界情况下会出现“超距仍可施法”或“在距但判超距”。

### D. 延时法术触发

- 玩法文档：轮结算中处理延时法术并消减寿命。
- 后端当前：已实现延时队列与寿命递减、到点触发、移除流程。

结论：延时主流程基本具备，但效果细节仍需补齐（如命中公式、抗性、筛选规则一致性）。

### E. 资源消耗

- 玩法文档：法术成功后减少行动位和法术位（上限约束）。
- 后端当前：大体已实现（即时法术与延时入队时都扣资源）。

结论：资源链路基本可用。

## 4. 后续实现建议（面向开发）

1. 先统一技能法术命中/发动公式，避免 C# 与 Python 两套行为分裂。
2. 明确 spellPower 字段来源（法术模板固定值 or 施法者属性派生）。
3. 在 Damage 分支补上法术减伤口径（至少与 magic_resist 有关联）。
4. 修正范围检查语义为"施法者到施法中心/目标"，减少边界歧义。
5. 固定范围法术敌我筛选策略并写入日志（便于复盘）。
6. 延时法术触发采用动态判定（触发时重新检查区域内目标）。
